# 機能追加依頼: ブラインドとアンティのスケジュールを統合する

## 概要

現状 `PokerTable` はブラインドの上昇スケジュール(`blind_schedule: list[tuple[int, int]]`)と
アンティの上昇スケジュール(`ante_schedule: list[int]`)を**別々の引数・別々のレベルカウンタ**
(`blind_level` / `ante_level`)として持っています。

これを **「SB/BB/アンティ」を1レベルとする単一のスケジュール** に統合してほしいです。

## 動機

- mullhouse 側のレベルアップ処理(`_run_level_up_loop`)は、ブラインドとアンティを常に**同じタイマーで
  同時に**上昇させています。実運用では2つのスケジュールは常に同じタイミングで進む前提になっているのに、
  APIとしては独立した2つのリスト・2つのレベルカウンタとして扱われていて、実態と乖離しています。
- 2つのリストの長さが揃っていないと(例: ブラインドは8段階あるのにアンティは3段階しかない)、
  `blind_level` と `ante_level` がそれぞれ勝手に頭打ちになり、把握しづらい不整合状態になり得ます。
  実際に発生してもエラーにはならず、静かにズレるだけなのが気になっています。
- 呼び出し側(mullhouse の卓作成フォーム・ボットスクリプト)も「ブラインドスケジュール」「アンティ
  スケジュール」という2つの入力欄・2つのカンマ区切り文字列を、インデックスを揃えて管理する必要があり、
  入力側の手間・ミスの元になっています。1レベル=「SB/BB/アンティ」の組にできれば、
  `25/50/0,50/100/25,100/200/50` のような1本の文字列で表現でき、シンプルになります。

## 現状のAPI

```python
PokerTable(
    table_id: str,
    max_players: int = 6,
    small_blind: int = 25,
    big_blind: int = 50,
    timeout_seconds: int = 30,
    blind_schedule: list[tuple[int, int]] | None = None,
    ante_schedule: list[int] | None = None,
    rake_percent: float = 0.0,
    rake_cap: int | None = None,
    rake_min_pot: int | None = None,
    allow_rebuy: bool = True,
    fixed_buy_in: int | None = None,
)
```

```python
def level_up_blind(self) -> GameEvent: ...  # BLIND_LEVEL_UP イベント、blind_level を進める
def level_up_ante(self) -> GameEvent: ...   # ANTE_LEVEL_UP イベント、ante_level を進める
```

`GameState` 側も `blind_level: int` と `ante_level: int` の2フィールドを持っています。

## 提案するAPI

```python
PokerTable(
    table_id: str,
    max_players: int = 6,
    small_blind: int = 25,
    big_blind: int = 50,
    ante: int = 0,
    timeout_seconds: int = 30,
    level_schedule: list[tuple[int, int, int]] | None = None,  # [(small_blind, big_blind, ante), ...]
    rake_percent: float = 0.0,
    rake_cap: int | None = None,
    rake_min_pot: int | None = None,
    allow_rebuy: bool = True,
    fixed_buy_in: int | None = None,
)
```

- `blind_schedule` / `ante_schedule` を廃止し、`level_schedule: list[tuple[int, int, int]] | None` に統合。
  各要素は `(small_blind, big_blind, ante)` の3つ組で、1レベルを表す。
- `level_schedule` 省略時は、現行の `blind_schedule` 省略時と同様に
  `[(small_blind, big_blind, ante)]` の単一レベルとして扱う(新たに `ante: int = 0` 引数を追加し、
  `small_blind`/`big_blind` と同格の「レベル0のアンティ」として使う)。
- `level_up_blind()` と `level_up_ante()` を統合し、単一の `level_up()` メソッドにする。
  1レベル進めて SB/BB/アンティをまとめて更新し、`LEVEL_UP` イベント(1種類)を
  `payload={"level": ..., "small_blind": ..., "big_blind": ..., "ante": ...}` で返す。
- `GameState.blind_level` と `GameState.ante_level` の2フィールドを、単一の `GameState.level: int` に統合。

## 呼び出し側(mullhouse)への影響

`poker_domain` の変更が入り次第、mullhouse 側で以下を追随修正します(このタスクでは対応不要です):

- `backend/poker_api.py`: `CreateTableRequest.blind_schedule`/`ante_schedule` → `level_schedule`
- `backend/poker_service.py`: `create_table()` / `_run_level_up_loop()`(`level_up_blind()`+`level_up_ante()`の
  個別呼び出し → `level_up()`1回)/ `serialize_state()`(`blind_level`+`ante_level` → `level`)
- フロントエンド: `PokerLobby.tsx` の「ブラインドスケジュール」「アンティスケジュール」の2入力欄を
  「レベルスケジュール(SB/BB/アンティ,...)」の1入力欄に統合。`pokerTypes.ts` の型も追随
- `bots/table_bot.py`: `--ante-schedule` を `--level-schedule`(`"25/50/0,50/100/25,..."`形式)に変更

## 追加してほしいテスト

`tests/unit/test_table_lifecycle.py` に、既存のブラインド/アンティのレベルアップ系テストを踏襲する形で:

- `level_schedule` 未指定時に `small_blind`/`big_blind`/`ante` の単一レベルで初期化されること
- `level_schedule` 指定時、レベル0の値で初期化され、`level_up()` を呼ぶたびに SB/BB/アンティが
  まとめて次のレベルに進むこと
- 最終レベルに達した後に `level_up()` を呼んでも変化しない(現行の `level_up_blind`/`level_up_ante`
  と同じ据え置き挙動)こと
